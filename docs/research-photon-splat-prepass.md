# Research: Photon Splat Prepass for Blender Cycles
**Branch:** `research/photon-splat`  
**Status:** Active Research  
**Last Updated:** 2026-04-15

---

## The Problem, Precisely Stated

Cycles is a **unidirectional path tracer**. It fires rays from the camera, bounces
them around the scene, and tries to randomly connect to a light source. In a
scene like a basement lit only through a doorway, a camera ray that enters the
basement must:

1. Hit a surface
2. Scatter in a random hemisphere direction
3. Scatter again... and again...
4. Eventually find a path that goes back through the door, through the house,
   and to a window or sky

The probability of finding that path by chance is **very low**. Low probability =
high variance = noise. This is the fundamental indirect illumination problem.

---

## The Algorithm Landscape

### 1. Photon Mapping (Jensen 1996, 2001)

The first principled solution to this exact problem. Two-pass:

- **Pass 1 (Light Pass):** Emit photons from light sources. Trace them forward through the scene, storing their hit locations, colors, and directions in a **photon map** (kd-tree in 3D space).
- **Pass 2 (Camera Pass):** Trace rays from camera. At each surface hit, query the photon map for nearby photons. Average their contributions within a search radius. This becomes the irradiance estimate.

**Why it works for the basement:** Photons shot from the light source trivially
find the door (they're going *outward*). They pour through the door, bounce
around inside the basement, and deposit information in the photon map. Camera
rays then only need 1 bounce to find the map, instead of randomly stumbling back
to the source.

**Problems:**
- **Bias:** The radiance estimate at a surface point is an average over a finite
  radius kernel. As radius → 0, bias → 0, but variance → ∞. You need both
  enough photons AND small enough radius, which are competing demands.
- **Memory:** Storing millions of photons globally is expensive.
- **Not unbiased:** The result never fully converges to ground truth at any
  finite photon count.

**Reference:** Jensen, H.W. (2001). *Realistic Image Synthesis Using Photon Mapping.* A.K. Peters.

---

### 2. Progressive Photon Mapping (PPM) — Hachisuka et al. 2008

The key innovation that solved the memory problem. Instead of storing all
photons forever, PPM runs iterative photon passes and **progressively shrinks**
the search radius. The algorithm is *biased but consistent*: given infinite
passes, it converges to the correct answer.

**Algorithm:**
```
1. Camera Pass: Trace rays from camera to first non-specular hit.
   Store these "visible points" (position, normal, throughput).
2. Repeat N times:
   a. Emit M photons from lights
   b. For each photon, check visible points within current radius R
   c. Accumulate photon flux at nearby visible points
   d. Reduce R by factor α (typically α ≈ 0.7)
   e. Update flux estimate using accumulated counts
```

**Memory:** Only the visible points are stored (one per pixel, per camera sample).
Photons are discarded each pass. Memory is now bounded.

**Convergence:** As passes → ∞, R → 0 and photon density → ∞. The bias
(from finite R) shrinks while variance (from finite photons per pass) also
shrinks. The estimator is consistent.

**Reference:** Hachisuka, T., Ogaki, S., Jensen, H.W. (2008). *Progressive Photon Mapping.* ACM TOG 27(5). SIGGRAPH Asia 2008.

---

### 3. Stochastic Progressive Photon Mapping (SPPM) — Hachisuka & Jensen 2009

Extends PPM to handle distributed ray tracing effects (depth of field, motion
blur, glossy surfaces). The key insight: instead of per-pixel visible points,
use **shared statistics across a pixel's region**. This allows multiple pixel
samples to contribute to the same photon statistics.

**Practical relevance:** This is the version that handles glossy basements, not
just pure diffuse. Most scenes have at least somewhat glossy surfaces.

**Reference:** Hachisuka, T., Jensen, H.W. (2009). *Stochastic Progressive Photon Mapping.* ACM TOG 28(5). SIGGRAPH Asia 2009.

---

### 4. Vertex Connection and Merging (VCM) — Georgiev et al. 2012

The current gold standard for offline rendering. Combines bidirectional path
tracing (BPT) *and* photon mapping under one mathematically unified framework
using **Multiple Importance Sampling (MIS)**.

**Two operations:**
- **Vertex Connection (VC):** Connect camera subpath vertices directly to light
  subpath vertices via shadow rays. This is classic BDPT.
- **Vertex Merging (VM):** At non-specular vertices, merge (photon map lookup)
  nearby light and camera subpath vertices. This is progressive photon mapping.

**MIS combines both:** Each path is weighted by the probability it could have
been generated by any of the available sampling strategies. This is provably
optimal combination.

**Why it dominates:** VCM handles all known difficult transport paths:
- Long indirect bounces (VM handles these)
- Caustics (VC handles these, VM helps)
- SDS paths (VC handles specular-diffuse-specular paths)
- Directly lit diffuse surfaces (VC, classic)

**Reference implementation:** SmallVCM — https://github.com/SmallVCM/SmallVCM
~1000 lines of C++, compares 7 algorithms. Indispensable.

**Reference:** Georgiev, I., Křivánek, J., Davidovič, T., Slusallek, P. (2012).
*Light Transport Simulation with Vertex Connection and Merging.* ACM TOG 31(6). SIGGRAPH Asia 2012.

Also see independently derived equivalent: **Unified Path Sampling (UPS)** —
Hachisuka, T., Pantaleoni, J., Jensen, H.W. (2012). *A Path Space Extension
for Robust Light Transport Simulation.* ACM TOG 31(6).

---

### 5. ReSTIR — Bitterli et al. 2020

Not photon mapping. A completely different approach using **Reservoir-based
Spatiotemporal Importance Resampling**. Designed for GPU real-time rendering.

**Core idea:**
1. For each pixel, generate many cheap candidate light samples
2. Select the best via weighted reservoir sampling (WRS) — only one sample
   survives per pixel
3. **Spatially reuse:** Neighboring pixels share their survivors via MIS
4. **Temporally reuse:** Previous frame's survivors are kept in a reservoir
   and reused

**Why it matters for noisy rooms:** A pixel near the door quickly finds a good
light path. Pixels deeper in the basement *borrow* that path sample. Information
propagates across the image.

**ReSTIR GI (2021):** The original ReSTIR was direct lighting only. ReSTIR GI
extends it to multi-bounce indirect. Ouyang et al. (2021) showed you can
resample full indirect paths, not just direct shadows.

**World-Space ReSTIR (SIGGRAPH Asia 2021):** Stores reservoirs in a 3D world-space
hash grid (not screen space), enabling reservoir reuse at arbitrary bounce depths.
This is the version most relevant to our addon concept — it's essentially
building a 3D spatial cache of good light paths.

**Status in Cycles:** Not implemented. Active research area. Partially overlaps
with path guiding conceptually.

**References:**
- Bitterli, B., et al. (2020). *Spatiotemporal Reservoir Resampling for Real-Time Ray Tracing.* ACM TOG 39(4).
- Ouyang, Y., et al. (2021). *ReSTIR GI: Path Resampling for Real-Time Path Tracing.* CGF 2021.
- SIGGRAPH Asia 2021. *World-Space Spatiotemporal Reservoir Reuse for Ray-Traced GI.*

---

### 6. Path Guiding — Intel OpenPGL (Cycles 3.4+)

**Already in Cycles.** Uses Intel's Open Path Guiding Library.

During rendering, Cycles builds a spatial approximation of the **radiance
distribution** — where light is actually coming from at each point in the scene.
This is learned online during the first N "training" samples. Subsequent samples
are biased toward high-radiance directions instead of sampling uniformly.

For the basement: the guiding structure learns that light comes from the door
direction, and future rays at basement surfaces are preferentially aimed there.

**Limitations:**
- CPU only (as of Blender 4.x — GPU support is on the roadmap, Blender Projects #92571)
- Requires a training period (128-256 samples). Bad ratio if total samples < 500.
- Not designed as a caustics solver
- Cannot be influenced externally from Python

**Key insight for our work:** Path guiding addresses the same noise problem but
from the *sampling strategy* direction. Our Photon Splat Prepass addresses it
from the *scene illumination* direction. These are complementary, not competing.
The Prepass reduces the *need* for guiding by making the light sources easier to
find in the first place.

## The Photon Splat Prepass: Full Technical Design

### Concept

We cannot do real photon mapping from a Python addon. But we can do a
**manual approximation** that achieves the same structural goal:

> *Light information from the door, captured during a prepass, is injected
> back into the scene as a glowing emission volume. The final render's camera
> rays need only 1-2 bounces to reach this volume, instead of the many bounces
> needed to trace back to the original source.*

This is analogous to what Lumen (Unreal Engine 5) does with its world-space
radiance cache, but done as a precompute in Blender Python.

---

### Stage 1: Interior Volume Probe Grid

**Goal:** Place probe positions inside the scene's interior space.

**Method — Parity Ray Cast (inside/outside test):**
```python
from mathutils.bvhtree import BVHTree
from mathutils import Vector

def is_inside(bvh, point):
    direction = Vector((1, 0, 0))   # arbitrary axis
    count = 0
    origin = point
    while True:
        hit, loc, normal, idx = bvh.ray_cast(origin, direction)
        if hit is None:
            break
        count += 1
        origin = loc + direction * 1e-5  # step past hit
    return (count % 2) == 1  # odd = inside
```

Cast once from the BVHTree built from the scene's enclosing mesh. Parity of
hits = inside or outside. This is O(depth of interior) per probe point — fast.

**Grid generation:**
- Get scene bounding box (union of all mesh bboxes)
- Sample `n×n×n` grid across bounding box
- Test each candidate position against the interior
- Keep only interior positions

**Density consideration:** Sparse grids give blocky caches. Dense grids slow
the prepass. A reasonable v1 default: 8×8×4 (XYZ). User-configurable.

**Alternative placement strategies:**
- Jittered grid (avoids regular grid artifacts)
- Adaptive density (cluster probes near bright/complex regions)
- Surface-seeded (Jones & Reinhart 2014 approach: place probes along actual
  surface paths traced from the camera)

### Stage 2: Irradiance Capture via Mini-Renders

**Goal:** At each probe position, determine the incoming radiance from all
directions.

**Method:**
```python
import bpy, numpy as np

def capture_probe_irradiance(probe_pos, samples=8, res=32):
    # Place equirectangular panoramic camera at probe position
    cam_data = bpy.data.cameras.new("ProbeCamera")
    cam_data.type = 'PANO'
    cam_data.cycles.panorama_type = 'EQUIRECTANGULAR'
    cam_obj = bpy.data.objects.new("ProbeCamera", cam_data)
    bpy.context.scene.collection.objects.link(cam_obj)
    cam_obj.location = probe_pos
    
    # Minimal render settings
    scene = bpy.context.scene
    orig_cam = scene.camera
    orig_res_x = scene.render.resolution_x
    orig_res_y = scene.render.resolution_y
    orig_samples = scene.cycles.samples
    
    scene.camera = cam_obj
    scene.render.resolution_x = res
    scene.render.resolution_y = res // 2  # 2:1 equirect
    scene.cycles.samples = samples
    
    bpy.ops.render.render()
    
    pixels = np.array(bpy.data.images['Render Result'].pixels[:])
    rgba = pixels.reshape(-1, 4)
    avg_rgb = rgba[:, :3].mean(axis=0)   # average over all directions
    
    # Restore
    scene.camera = orig_cam
    scene.render.resolution_x = orig_res_x
    scene.render.resolution_y = orig_res_y
    scene.cycles.samples = orig_samples
    bpy.data.objects.remove(cam_obj)
    bpy.data.cameras.remove(cam_data)
    
    return avg_rgb
```

**Key design choices:**
- `32×16` at `8 samples` = ~1/1000th of a normal render's cost, per probe
- 400 probes = ~0.4 equivalent renders. Fast.
- Average all pixel colors → scalar irradiance estimate at that point
- No directional information is preserved (yet — see Future Work)

**What this loses:** Directionality. The probe returns average color, not
which direction the light came from. This means the injected volume cannot
provide directional shading — only ambient fill.

**For v1, this is acceptable.** The goal is reducing noise, not matching
directional lighting exactly.

**Directional improvement path (v2):**
Instead of averaging all pixels, store the full equirectangular image per probe
as a **spherical function** encoded in Spherical Harmonics (SH). Each probe then
stores 9 SH coefficients (L0, L1, L2) per color channel = 27 floats. This is
exactly what Lumen's screen probes and Eevee's irradiance volumes do.

### Stage 3: VDB Construction

**Goal:** Convert probe positions + irradiance colors into a 3D OpenVDB volume.

**pyopenvdb availability:**
- Bundled with official Blender distributions from v3.6+
- Not guaranteed in distro-installed builds (Linux package managers)
- Check: `import pyopenvdb` in Blender Python Console

**Grid structure:**
```python
import pyopenvdb as vdb
import numpy as np

def build_irradiance_vdb(probe_data, bbox, voxel_size, output_path):
    """
    probe_data: list of (world_pos, rgb_color) tuples
    bbox: (min_xyz, max_xyz)
    voxel_size: world units per voxel
    """
    # Emission grid (Vec3SGrid = RGB float vector)
    emission_grid = vdb.Vec3SGrid(background=(0.0, 0.0, 0.0))
    emission_grid.name = 'emission'
    emission_grid.gridClass = vdb.GridClass.FOG_VOLUME
    
    # Density grid (very low — just enough for volume sampling)
    density_grid = vdb.FloatGrid(background=0.0)
    density_grid.name = 'density'
    density_grid.gridClass = vdb.GridClass.FOG_VOLUME
    
    em_acc = emission_grid.getAccessor()
    de_acc = density_grid.getAccessor()
    
    origin = bbox[0]
    
    for world_pos, rgb in probe_data:
        # Convert world position to voxel index
        ix = int((world_pos[0] - origin[0]) / voxel_size)
        iy = int((world_pos[1] - origin[1]) / voxel_size)
        iz = int((world_pos[2] - origin[2]) / voxel_size)
        
        em_acc.setValueOn((ix, iy, iz), tuple(float(c) for c in rgb))
        de_acc.setValueOn((ix, iy, iz), 0.01)  # minimal density
    
    # Critical: smooth the sparse probe data into a continuous volume
    # A Gaussian blur over the voxel array fills gaps between probes
    emission_grid = gaussian_blur_vdb(emission_grid, radius=2)
    
    vdb.write(output_path, grids=[emission_grid, density_grid])

def gaussian_blur_vdb(grid, radius=2):
    """Blur via numpy → copyToArray → convolve → copyFromArray"""
    import scipy.ndimage
    
    # Get dense numpy array
    arr = np.zeros((*get_grid_dims(grid), 3), dtype=np.float32)
    grid.copyToArray(arr)
    
    # Gaussian blur per channel
    for c in range(3):
        arr[..., c] = scipy.ndimage.gaussian_filter(arr[..., c], sigma=radius)
    
    # Write back
    out_grid = vdb.Vec3SGrid(background=(0.0, 0.0, 0.0))
    out_grid.name = grid.name
    out_grid.gridClass = grid.gridClass
    out_grid.copyFromArray(arr)
    return out_grid
```

**The blur is critical:** Without it, the volume is spotty — only the exact
probe voxels are lit. Gaussian blur propagates probe values to neighboring
voxels, creating a smooth, continuous radiance field. The radius controls how
far a single probe's influence extends.

**Note on `scipy`:** scipy may not be bundled in Blender Python. Fallback:
implement a simple 3D box blur in pure numpy (slower but dependency-free).

### Stage 4: Volume Injection and Shader

**Goal:** Import the VDB and assign a pure emission shader.

```python
def inject_irradiance_volume(vdb_path, emission_strength=1.0):
    # Add volume object
    bpy.ops.object.volume_add()
    vol_obj = bpy.context.active_object
    vol_obj.name = "IrradianceCache"
    vol_obj.data.filepath = vdb_path
    
    # Create pure emission material
    mat = bpy.data.materials.new("IrradianceCache_Mat")
    mat.use_nodes = True
    nodes = mat.node_tree.nodes
    links = mat.node_tree.links
    
    nodes.clear()
    
    # Attribute node to read emission grid
    attr = nodes.new('ShaderNodeAttribute')
    attr.attribute_name = 'emission'
    attr.attribute_type = 'GEOMETRY'
    
    # Emission shader
    emission = nodes.new('ShaderNodeEmission')
    emission.inputs['Strength'].default_value = emission_strength
    
    # Volume output
    output = nodes.new('ShaderNodeOutputMaterial')
    
    links.new(attr.outputs['Color'], emission.inputs['Color'])
    links.new(emission.outputs['Emission'], output.inputs['Volume'])
    
    vol_obj.data.materials.append(mat)
    return vol_obj
```

**Why pure emission (no scatter, no absorption):**
- Camera rays pass through the volume without being attenuated
- The volume is transparent to all primary rays
- It only contributes when a **bounce ray** hits it
- Bounce rays that would have continued for 3+ more hops instead hit this
  glowing volume and terminate early
- This is exactly the short-circuit we want

**Why this doesn't block light:**
The volume has no `Principled Volume` node, no density-driven absorption.
Camera rays shoot straight through. Only scattered secondary rays "see" the
emission. This is the key physical design.

### The Double-Counting Problem

This is the most critical correctness issue in the design.

**The problem:**
The emission volume was baked *from* the scene's lighting. If we then render
the scene *with* the volume, and also allow Cycles' own indirect paths to
resolve, we count the indirect light contribution twice:

```
Indirect Light = (Cycles's indirect paths) + (Volume's baked indirect)
                                           ↑ double counted
```

**Solution strategies:**

**Strategy A — Bake direct-only, render indirect-only:**
```
Prepass: Set max bounces = 1 (direct light only). Bake probes.
Final Render: Set max bounces = 1. Enable volume.
              + Volume carries all multi-bounce light
              = No overlap. Volume fills exactly the indirect Cycles misses.
```
Clean. Correct. But Cycles max bounces = 1 means no color bleeding from
1-bounce indirect, which is often quite visible. The volume compensates.

**Strategy B — Bake at N bounces, render at N-1 bounces:**
```
Prepass: Set max bounces = N. Bake probes capturing full indirect.
Final Render: Set max bounces = N-1. Enable volume.
              Volume fills the Nth bounce and beyond.
              Cycles handles up to (N-1) bounces.
              = Small overlap at exactly the Nth bounce only.
```
The overlap is controlled and small. For N=4 this is practical.

**Strategy C — Light Path Expression masking (future):**
Cycles has Light Path Expressions (LPEs) which could theoretically isolate
the contribution of paths beyond N bounces. However, there's no Python API to
inject LPE-conditional shaders programmatically. This would require Cycles
integrator modifications.

**Strategy D — Volume emission strength scaling:**
Empirically calibrate the volume emission strength so that when it *adds* to
Cycles' remaining indirect bounces, the total matches a ground truth render.
This is a cheat but works visually.

**Recommended for v1: Strategy A.** Clear, no approximation.
**Recommended for v2: Strategy B with N=4.** Better indirect quality.

## What Related Systems Do

### Lumen (Unreal Engine 5)
Lumen uses a **world-space radiance cache** that is the closest real production
system to what we're building. Key architectural points:
- **MeshCards:** Scene surfaces are represented as simplified planar cards
  (6 orientations per mesh). Cards store direct+indirect lighting baked into
  a surface cache atlas texture.
- **Screen Probes:** A low-resolution grid of probes placed per-pixel in screen
  space. Each probe traces 64 rays into the radiance cache.
- **Final Gather:** Screen probes integrate nearby irradiance and upsample to
  full resolution.
- **World-Space Radiance Cache:** For probes beyond ~100m or for secondary
  bounces, voxel lighting via Global SDF is used.

Our addon is doing a simplified version: no MeshCards, no SDF, just:
equirect captures → VDB → emission volume.

### EEVEE Next Irradiance Volume (Blender 4.x)
EEVEE's irradiance volume baking (Blender Projects #108639) works via:
1. **Surfel placement:** A rasterization-based pass that scatters tiny surface
   elements (surfels) across the scene geometry
2. **Light propagation:** Photon-like passes that accumulate direct light at
   surfels, then propagate to neighbors iteratively
3. **SH encoding:** Per-probe Spherical Harmonics coefficients, stored in a
   3D texture atlas

We can *reuse EEVEE's baked irradiance data* in Cycles by reading probe SH
values and injecting them into a VDB. This would be a v2 approach — skip the
mini-render loop entirely and use EEVEE's faster bake infrastructure.

## Blender Python API: What's Available

### Render Hooks
```python
bpy.app.handlers.render_pre    # fires before render starts
bpy.app.handlers.render_post   # fires after render completes
bpy.app.handlers.render_complete  # fires on successful completion
```
The prepass can be triggered automatically from `render_pre`. Lock the
interface during render: `scene.render.use_lock_interface = True`.

### Scene Raycast
```python
result, location, normal, index, obj, matrix = bpy.context.scene.ray_cast(
    depsgraph,
    origin,    # Vector
    direction  # Vector
)
```
World-space ray cast against all objects. Used for interior detection via
parity test.

### BVH Tree (faster for millions of tests)
```python
from mathutils.bvhtree import BVHTree
bvh = BVHTree.FromObject(obj, depsgraph)
hit, loc, normal, idx = bvh.ray_cast(origin, direction)
```
Cached. Use for interior probe placement when testing thousands of candidate points.

### pyopenvdb
- Available: Blender 3.6+ (official builds)
- Grid types: `FloatGrid`, `Vec3SGrid`, `BoolGrid`
- Critical methods: `getAccessor()`, `setValueOn()`, `copyFromArray()`,
  `copyToArray()`, `gridClass = FOG_VOLUME`
- Write: `vdb.write(path, grids=[g1, g2])`
- World transform: `grid.transform = vdb.createLinearTransform(voxel_size)`

### Volume Object Creation
```python
bpy.ops.object.volume_add()
obj.data.filepath = '/path/to/file.vdb'
```
VDB filepath can be updated at runtime — addon can rebake and the object
auto-refreshes.

## Known Unknowns and Open Questions

### 1. pyopenvdb's `copyFromArray` for Vec3SGrid
The PBRT and pyopenvdb docs confirm `copyFromArray` works for Vec3SGrid with a
4D numpy array `(x, y, z, 3)`. However, I have not verified this works inside
Blender's bundled pyopenvdb. The BVTKNodes addon uses manual accessor loops as
a fallback. Need to test both paths.

**Confidence:** ~75% that `copyFromArray` works. Fallback is the accessor loop.

### 2. scipy.ndimage availability
scipy is not bundled with Blender Python. Options:
- Implement manual 3D gaussian blur in numpy (separable 1D convolutions → 3D)
- Install scipy into Blender's Python: `bpy.ops.preferences.addon_install` or
  subprocess pip install into Blender's site-packages
- Use OpenVDB's built-in level set filters (more complex)

**Recommended:** Pure numpy blur for zero dependency requirement.

### 3. Mini-render loop overhead
Each probe requires: camera setup, render op, pixel readback, camera cleanup.
The `bpy.ops.render.render()` call has Python overhead per invocation (~0.1s?).
400 probes × 0.1s = 40s overhead on top of actual render time.

**Mitigation:** Batch renders using render animation with a probe camera rig.
Or use a custom render engine override for the prepass. Needs benchmarking.

### 4. Persistent Volume Object
The volume's filepath needs to survive file saves and reopens. Using a path
relative to the `.blend` file with `bpy.path.abspath` is standard practice.
Need to handle the case where the VDB was baked in a temp dir and the file
was moved.

### 5. Light Leaking Through Walls
The pure emission volume has no awareness of geometry. A probe inside a wall
(if interior detection fails) will inject light that bleeds through surfaces.
This is the same light leak problem Eevee's irradiance volumes suffer from.

**Mitigation:** Bias probe positions along the normals of nearby surfaces
(move them slightly toward the interior). The Jones & Reinhart (2014) paper
addresses exactly this.

### 6. Double-Counting Calibration
Strategy A (direct-only bake) avoids the problem but may underestimate
ambient fill quality. The emission strength slider is the user-facing knob,
but the "correct" value is scene-dependent. A calibration step (compare
a small region with reference) could automate this.

## Proposed Addon Architecture

```
PhotonSplatPrepass/
    __init__.py             ← register, UI panel
    probe_grid.py           ← Stage 1: interior point generation + BVH tests
    capture.py              ← Stage 2: mini-render loop per probe
    vdb_builder.py          ← Stage 3: pyopenvdb grid construction + blur
    injector.py             ← Stage 4: volume object + emission shader
    calibration.py          ← double-counting compensation logic
    ui.py                   ← sidebar panel
    utils.py                ← shared helpers (bbox, world/voxel coords)
    session_notes.md        ← per-session dev notes
```

**UI Panel (N-panel → "Photon Splat"):**
- `[Bake Irradiance Cache]` button → triggers full pipeline
- `[Clear Cache]` button → removes volume object + VDB file
- Grid Resolution slider (default: 8×8×4)
- Capture Samples (default: 8)
- Emission Strength (default: 1.0)
- Bake Bounces (default: 1, for Strategy A)
- Status readout: "Ready / Baking probe 12/64 / Done (64 probes)"
